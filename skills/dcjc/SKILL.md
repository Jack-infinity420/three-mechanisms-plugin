# 督促检查

## 自我审查清单

每完成一个工作单元，逐项检查：

- **代码质量**：语法正确、逻辑清晰、边界处理完善、无冗余或过度抽象
- **工具调用**：调用正确、必要、无遗漏
- **进度跟踪**：是否按计划推进、有无延期风险
- **资源使用**：计算、存储、上下文窗口使用是否合理

## 审查频率

| 任务类型 | 审查频率 |
|---------|---------|
| 常规任务 | 每里程碑一次 |
| 复杂任务 | 每子任务一次 |
| 紧急任务 | 实时监控 |

## 异常处理

发现执行偏离计划、代码质量问题、工具错误、协作异常时：

```
告警 → 分析 → 修正 → 验证
```

1. **告警**：立即向用户报告
2. **分析**：定位根因，明确范围
3. **修正**：给出修复方案并整改
4. **验证**：确认解决，防止复发

## 进度推动

- 进度滞后：主动提醒，分析原因，提出追赶方案
- 遇到阻塞：自行可解的立即处理，需协调的升级上报

## 告警格式

```
⚠️ 质量告警
━━━━━━━━━━━━━━━━━━━━
【问题类型】：代码质量 / 进度偏差 / 工具异常 / 协作问题
【严重程度】：高 / 中 / 低
【问题描述】：具体问题说明
【影响范围】：受影响的模块或任务
【建议方案】：修复建议
【处理状态】：待处理 / 处理中 / 已解决
━━━━━━━━━━━━━━━━━━━━
```

## 理性化对照表

跳过审查或隐匿问题的常见借口：

| 借口 | 实际 |
|------|------|
| "这改动太简单，不需要审查" | 简单改动引入 bug 的案例最多。30 秒自检 > 数小时修复 |
| "问题不大，我修完再说" | 自行消化问题 = 延误处理窗口。用户可能是最快解围的人 |
| "报告了也没用" | 隐瞒问题一定比报告问题更糟。用户有权知道 |
| "审查会拖慢速度" | 代码回滚和线上修复比审查慢得多 |
| "前面都正常，这次也不会有问题" | 累积偏差最危险。每次都要检查 |

## Red Flags — 立即触发质量告警

- 核心逻辑存在缺陷，可能导致功能错误
- 进度延误超过预估时间 50%
- 工具调用连续失败 3 次以上
- 发现安全漏洞或数据丢失风险
- 任务依赖关系被破坏
- "这个问题我自己能处理"
- "先跳过审查，后面再补"
- "等任务做完再统一检查"

**以上任一信号 = 立即输出告警模板，不得自行消化。**

## 常见错误

- 仅任务结束时检查，错过中期修正窗口
- 发现问题自行消化不报告，延误处理
- 审查流于形式，忽略深层逻辑
- 告警信息笼统，缺少定位和方案
- 过度审查拖慢效率，应按任务复杂度调节深度
