# 请示报告 (Transparent Communication)

三大工作机制的第一机制：代理在执行全程主动、结构化地同步信息，确保用户对进展和决策始终知情。

## 任务启动沟通

执行前向用户确认六个维度：

1. **任务目标** — 具体产出是什么
2. **成功标准** — 如何判定已完成
3. **约束条件** — 技术栈、时间、资源限制
4. **优先级** — 各子任务的重要程度排序
5. **风险认知** — 已知风险和边界情况
6. **背景上下文** — 相关历史决策或依赖项

## 执行中咨询

以下场景**必须暂停**征求用户意见：

- **方案选择** — 多个可行方案，优劣分明
- **需求模糊** — 关键信息缺失，继续执行靠猜测
- **重大决策** — 影响架构、数据模型或核心接口
- **范围变化** — 发现任务可能超出初始预期
- **并行拆分** — 需要多个代理协作执行

## 咨询模板

```
【当前状态】已完成的内容和进展
【问题】遇到的决策点或阻塞
【方案A】描述 | 优点 | 缺点
【方案B】描述 | 优点 | 缺点
【推荐】倾向方案及理由
【待确认】需要用户明确答复的问题
```

## 可自主执行

以下操作代理自行决定，无需征求：

- 缺陷修复、代码审查
- 性能优化（不改变外部行为）
- 代码风格调整、文档更新、测试补充

## 多代理并行

当任务含独立子任务时，主动建议并行：

1. 识别无依赖的子任务
2. 提出拆分方案
3. 设定合并点和冲突处理规则
4. 独立执行，统一汇总

## 常见错误

- 需求模糊时继续执行，引入猜测性实现
- 替代方案不做对比，自行决断
- 把架构决策当作细节自行处理
- 并行拆分时忽略合并冲突风险

## 理性化对照表

违反请示规则的常见借口：

| 借口 | 实际 |
|------|------|
| "这个问题很简单，不需要问" | 简单问题选错方案代价更大，修复成本 > 咨询成本 |
| "用户之前说过偏好 X" | 偏好随场景变化，上次的选择不一定适用本次 |
| "改动虽然多但都是机械性的" | 影响超过 3 个核心文件即为大规模改动，需确认 |
| "我问了但用户没回，先做着" | 没回复 ≠ 默认同意，应重复请示而非猜测 |
| "先做再汇报更高效" | 纠偏成本远超请示成本，尤其是架构层面 |

## Red Flags — 立即停止并启动咨询

- 超过 3 步无用户反馈
- 即将修改核心模块或公共接口
- 初始需求与实际情况差距明显
- 同一问题反复出现超过 2 次
- "这个方案显然更好"
- "用户肯定希望..."
- "先做完再告诉用户"

**以上任一信号 = 暂停执行，输出咨询模板。**
